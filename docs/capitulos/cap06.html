6.1 Introdução

O capítulo anterior construiu layouts a partir do zero — box model, floats, Flexbox, Grid. Este capítulo parte dessas fundações e introduz uma camada de abstração: os frameworks CSS, bibliotecas que encapsulam padrões de layout e componentes em classes reutilizáveis.

O Bootstrap é o framework mais utilizado no mundo e o escolhido nesta UC. Mas antes de o usar, é essencial compreender o que ele faz por baixo — a grelha de 12 colunas que estudámos no capítulo anterior, as media queries, os modelos Flexbox — porque essa compreensão é o que distingue quem usa um framework com domínio de quem apenas copia classes.

A segunda metade do capítulo aborda as micro-interações em CSS puro: transições, transformações e animações. O Bootstrap usa estas técnicas internamente; aprenderem-nas em CSS nativo significa levá-las para qualquer projecto, com ou sem framework.

6.2 6.1 O Problema que os Frameworks Resolvem

6.2.1 6.1.1 CSS manual a escala

Num projecto pequeno, escrever CSS do zero é rápido e directo. A partir de um certo ponto, a escala impõe três problemas estruturais:

Consistência — manter os mesmos espaçamentos, tamanhos e cores em dezenas de componentes (botões, formulários, navbars) exige repetição manual. Uma alteração ao estilo de um botão pode implicar editar centenas de linhas em múltiplos ficheiros.

Responsividade — escrever @media queries para todos os breakpoints de cada componente é trabalhoso. Vinte componentes com quatro breakpoints cada equivale a oitenta blocos de media queries para manter.

Manutenção — o CSS não tem sistema de tipos nem módulos nativos. CSS não estruturado cresce para um estado que os programadores designam por CSS soup: um ficheiro difícil de ler, difícil de alterar sem quebrar algo inesperado.

6.2.2 6.1.2 O que é um framework CSS

Um framework CSS é uma biblioteca de estilos, componentes e convenções prontas a usar — escritas uma vez, reutilizáveis em qualquer projecto. Em vez de escrever o CSS de cada botão ou navbar do zero, o framework fornece classes que se aplicam directamente no HTML.

O compromisso é claro: ganha-se velocidade e consistência, perde-se algum controlo fino sobre o CSS gerado. Na maioria dos projectos, este compromisso é favorável.

6.3 6.2 Bootstrap — Instalação via CDN

6.3.1 6.2.1 O que é um CDN

Um CDN (Content Delivery Network) é uma rede distribuída de servidores que disponibiliza ficheiros estáticos (CSS, JavaScript, fontes) a partir do nó geograficamente mais próximo do utilizador. Usar o Bootstrap via CDN significa não descarregar nem hospedar os ficheiros — o browser carrega-os directamente dos servidores do Bootstrap.

Vantagens:

  • Configuração imediata — sem instalação, sem npm, sem build step
  • O ficheiro pode estar em cache no browser do utilizador (se já visitou outro site com o mesmo CDN)
  • Sempre actualizado para a versão especificada

Desvantagens:

  • Dependência de conectividade — não funciona offline
  • Impossível personalizar (alterar variáveis SASS, etc.)
  • Produção: preferível servir os ficheiros do próprio servidor

6.3.2 6.2.2 Estrutura mínima de uma página Bootstrap

Uma página HTML com Bootstrap requer três elementos essenciais:

<!DOCTYPE html>
<html lang="pt">
<head>
  <meta charset="UTF-8">

  <!-- 1. viewport meta — essencial para responsividade -->
  <meta name="viewport" content="width=device-width, initial-scale=1">

  <!-- 2. Bootstrap CSS — deve vir antes de qualquer CSS personalizado -->
  <link
    href="https://cdn.jsdelivr.net/npm/bootstrap@5.3.3/dist/css/bootstrap.min.css"
    rel="stylesheet">

  <title>Página com Bootstrap</title>
</head>
<body>

  <!-- conteúdo aqui -->
  <div class="container">
    <h1>Olá, Bootstrap!</h1>
  </div>

  <!-- 3. Bootstrap JS Bundle — no final do body, antes de </body> -->
  <!-- inclui Popper.js para tooltips, dropdowns e modals -->
  <script
    src="https://cdn.jsdelivr.net/npm/bootstrap@5.3.3/dist/js/bootstrap.bundle.min.js">
  </script>

</body>
</html>
CSS antes do JS

O CSS deve ser incluído no <head> para que o estilo seja aplicado antes de o conteúdo ser renderizado. O JS é colocado no final do <body> para não bloquear o carregamento da página. O bootstrap.bundle.min.js inclui já o Popper.js — não é necessário adicioná-lo separadamente.

6.3.3 6.2.3 CSS personalizado sobre Bootstrap

Ao adicionar CSS personalizado, este deve ser incluído depois do CSS do Bootstrap para que as regras próprias possam sobrepor as do framework:

<head>
  <!-- 1. Bootstrap CSS -->
  <link href="bootstrap.min.css" rel="stylesheet">

  <!-- 2. CSS próprio — vem depois, tem precedência -->
  <link href="styles.css" rel="stylesheet">
</head>

6.4 6.3 O Sistema de Grid Bootstrap

6.4.1 6.3.1 Container, row e col

O grid Bootstrap assenta em três classes hierárquicas que espelham exactamente a grelha de 12 colunas estudada no capítulo anterior — a diferença é que o Bootstrap gere o clearfix, os gutters e a responsividade automaticamente.

<div class="container">         <!-- centra e define largura máxima -->
  <div class="row g-3">         <!-- linha de grelha; g-3 = gutter de 1rem -->
    <div class="col-8">         <!-- ocupa 8 das 12 colunas -->
      Conteúdo principal
    </div>
    <div class="col-4">         <!-- ocupa 4 das 12 colunas -->
      Sidebar
    </div>
  </div>
</div>

As colunas da mesma row devem somar 12. A classe col (sem número) distribui o espaço restante igualmente entre todos os col da linha.

6.4.2 6.3.2 Os 6 breakpoints do Bootstrap

O Bootstrap define seis pontos de quebra, todos com min-width — confirmando que o Bootstrap é mobile-first por definição. Abaixo de 576px não existe media query; o CSS base aplica-se a todos os tamanhos.

Nome Prefixo Condição Container
xs (nenhum) < 576px — sem query 100%
sm sm ≥ 576px 540px
md md ≥ 768px 720px
lg lg ≥ 992px 960px
xl xl ≥ 1200px 1140px
xxl xxl ≥ 1400px 1320px
Container ≠ Breakpoint

O container é sempre mais estreito que o breakpoint — para md (≥ 768px), o container tem 720px. Os 48px restantes (24px de cada lado) formam as margens automáticas que centram o conteúdo. O container-fluid é a excepção: ocupa sempre 100% da largura.

6.4.3 6.3.3 Classes responsivas — col-{bp}-{n}

A responsividade Bootstrap é declarativa: especifica-se no HTML como cada coluna se comporta a cada breakpoint, sem escrever uma única media query:

<!-- Uma coluna que se adapta automaticamente:
     mobile (< 768px): linha completa
     tablet (≥ 768px): meia largura
     desktop (≥ 992px): um terço da largura -->

<div class="row g-3">
  <div class="col-12 col-md-6 col-lg-4">Card A</div>
  <div class="col-12 col-md-6 col-lg-4">Card B</div>
  <div class="col-12 col-md-6 col-lg-4">Card C</div>
</div>

As classes aplicam-se de forma cumulativa — col-md-6 aplica-se em md e em todos os breakpoints maiores, até ser sobreposta por col-lg-4.

6.6 6.5 Utilitários Bootstrap

Os utilitários são classes atómicas que substituem CSS de uma linha — aplicadas directamente no HTML, sem escrever regras adicionais.

6.6.1 6.5.1 Spacing — margin e padding

O padrão é {tipo}{lado}-{escala}, onde o tipo é m (margin) ou p (padding), o lado é opcional (t, b, s, e, x, y), e a escala vai de 0 a 5 (em múltiplos de 0.25rem) ou auto.

<div class="mt-3">   <!-- margin-top: 1rem -->
<div class="mb-2">   <!-- margin-bottom: 0.5rem -->
<div class="px-4">   <!-- padding horizontal: 1.5rem -->
<div class="py-2">   <!-- padding vertical: 0.5rem -->
<div class="m-0">    <!-- margin: 0 em todos os lados -->
<div class="mx-auto"> <!-- margin horizontal: auto → centrado -->

6.7 6.6 Micro-interações em CSS Puro

6.7.1 6.6.1 O Bootstrap já anima — mas como?

Se inspeccionarmos o CSS do Bootstrap com as DevTools do browser (F12 → Sources → bootstrap.min.css), encontramos isto:

/* Bootstrap internals — transition nos botões */
.btn {
  transition: color .15s ease-in-out,
              background-color .15s ease-in-out,
              border-color .15s ease-in-out,
              box-shadow .15s ease-in-out;
}

/* Transição no collapse da navbar */
.collapsing { transition: height .35s ease; }

/* Fade nos modals */
.fade { transition: opacity .15s linear; }

O efeito de hover dos botões, a abertura suave da navbar, o fade dos modals — nada disto é JavaScript. É CSS puro. O Bootstrap escreve as transições por nós; as secções seguintes ensinam a fazê-lo por conta própria, em qualquer contexto, com ou sem framework.

6.7.2 6.6.2 transition

A propriedade transition suaviza a mudança de qualquer propriedade CSS ao longo do tempo. A sintaxe é:

transition: propriedade duração easing atraso;
/* Exemplo simples: mudar a cor de fundo em 0.3s */
.btn {
  background: #0d6efd;
  transition: background-color 0.3s ease;
}
.btn:hover { background: #0a58ca; }

/* Múltiplas propriedades */
.card {
  transform: translateY(0);
  box-shadow: none;
  transition: transform 0.2s ease,
              box-shadow 0.2s ease;
}
.card:hover {
  transform: translateY(-6px);
  box-shadow: 0 8px 24px rgba(0,0,0,.15);
}

/* ⚠ Evitar: transition: all — anima tudo, impacto de performance */

6.7.3 6.6.3 transform

A propriedade transform modifica a aparência visual de um elemento — movendo-o, escalando-o ou rodando-o — sem alterar o fluxo do documento. Os elementos vizinhos não se deslocam. O browser usa aceleração GPU, o que torna as animações com transform mais eficientes do que animar top ou left.

/* Mover */
.el:hover { transform: translateY(-8px); }   /* desloca para cima 8px */
.el:hover { transform: translateX(4px); }    /* desloca para a direita */
.el:hover { transform: translate(-4px, -4px); } /* ambos os eixos */

/* Escalar */
.el:hover { transform: scale(1.05); }        /* 5% maior */
.el:hover { transform: scale(0.95); }        /* 5% menor */

/* Rodar */
.arrow:hover { transform: rotate(90deg); }   /* sentido horário */
.icon.open   { transform: rotate(45deg); }   /* ícone × a partir de + */

/* Combinar — sempre numa única declaração */
.card:hover {
  transform: translateY(-4px) scale(1.02);   /* ✓ */
}
/* ✗ Errado: duas declarações transform sobrepoem-se */
.card:hover { transform: translateY(-4px); }
.card:hover { transform: scale(1.02); }      /* cancela a anterior */

6.7.4 6.6.4 Pseudo-classes como gatilhos

As pseudo-classes definem estados de um elemento — são o mecanismo que activa as micro-interações sem JavaScript.

.btn {
  background: #0d6efd;
  color: white;
  transition: background 0.2s ease, transform 0.1s ease;
}

/* Estado: rato sobre o elemento */
.btn:hover {
  background: #0a58ca;
  transform: translateY(-2px);
}

/* Estado: elemento pressionado */
.btn:active {
  transform: translateY(0);   /* "afunda" no clique */
}

/* Estado: elemento com foco (teclado ou clique) */
.btn:focus {
  outline: 3px solid rgba(13,110,253,.4);
  outline-offset: 2px;
}

/* Estado: foco apenas por teclado (não por clique) */
/* mais limpo que :focus para utilizadores de rato */
.btn:focus-visible {
  outline: 3px solid #0d6efd;
  outline-offset: 2px;
}
:focus e acessibilidade

Nunca remover o outline de foco sem fornecer uma alternativa visual. O outline é o único indicador de posição para utilizadores que navegam por teclado (pessoas com limitações motoras). O :focus-visible permite mostrar o outline apenas durante navegação por teclado, sem o mostrar a utilizadores de rato — o melhor dos dois mundos.

6.7.5 6.6.5 @keyframes — animações autónomas

Enquanto transition reage a uma mudança de estado (hover, focus), o @keyframes define uma sequência de estados que corre automaticamente, independentemente da interação do utilizador.

A utilização é um processo em dois passos:

Passo 1 — Definir a animação com @keyframes:

/* Sintaxe with from/to */
@keyframes fadeInUp {
  from {
    opacity: 0;
    transform: translateY(20px);
  }
  /* ponto intermédio opcional: */
  /* 50% { opacity: 0.5; }      */
  to {
    opacity: 1;
    transform: translateY(0);
  }
}

/* Sintaxe com percentagens — permite múltiplos estados */
@keyframes pulse {
  0%, 100% { transform: scale(1); }
  50%       { transform: scale(1.05); }
}

Passo 2 — Aplicar com a propriedade animation:

/* Shorthand: nome duração easing atraso iterações direção fill-mode */
.hero    { animation: fadeInUp 0.6s ease-out; }
.badge   { animation: pulse 2s ease-in-out infinite; }

/* Propriedades individuais */
.card {
  animation-name:            fadeInUp;
  animation-duration:        0.4s;
  animation-timing-function: ease-out;
  animation-delay:           0.1s;
  animation-fill-mode:       forwards; /* mantém o estado final */
}
animation-fill-mode: forwards

Por omissão, quando uma animação termina, o elemento volta ao estado inicial. fill-mode: forwards mantém o estado final da animação — essencial para efeitos de entrada como fadeInUp, onde queremos que o elemento permaneça visível depois de aparecer.

6.7.6 6.6.6 Boas práticas e acessibilidade

/* ✓ Especificar propriedades — não usar transition: all */
.btn { transition: background-color 0.2s ease, transform 0.1s ease; }

/* ✓ Preferir transform e opacity — aceleração GPU, sem reflow */
/* ✗ Evitar animar top, left, width, height — causam reflow */

/* ✓ Respeitar prefers-reduced-motion */
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration:   0.01ms !important;
    transition-duration:  0.01ms !important;
  }
}

/* ✓ Nunca remover :focus sem alternativa visível */
/* ✗ .btn:focus { outline: none; }  — inacessível */
/* ✓ .btn:focus-visible { outline: 3px solid #86b7fe; } */

/* ✓ Animar com propósito — cada micro-interação deve comunicar algo:
      hover → "isto é clicável"
      active → "o clique foi registado"
      fadeInUp → "novo conteúdo chegou"      */

6.8 Síntese do Capítulo

Os frameworks CSS resolvem o problema da consistência e da responsividade a escala. O Bootstrap é o mais adoptado: inclui via CDN, usa classes no HTML, e encapsula as media queries, o grid e os componentes mais comuns.

O grid Bootstrap é o sistema de 12 colunas do capítulo anterior, automatizado e responsivo. Os seis breakpoints (xs a xxl) usam todos min-width — o Bootstrap é mobile-first por definição. As classes col-{bp}-{n} aplicam-se a partir do breakpoint indicado e são cumulativas.

Os componentes (navbar, cards, buttons, forms) e os utilitários (spacing, display, flex, texto, cores) eliminam CSS repetitivo. A regra de ouro é incluir o CSS próprio depois do Bootstrap para poder sobrepor estilos.

As micro-interações — transition, transform, pseudo-classes e @keyframes — são CSS puro, independentes do Bootstrap. O Bootstrap usa-as internamente; aprendê-las em CSS nativo significa usá-las em qualquer projecto. transform e opacity são as propriedades a animar: usam GPU, não causam reflow, e produzem transições fluidas. Respeitar prefers-reduced-motion e nunca remover :focus sem alternativa são requisitos de acessibilidade, não opções.

## Introdução {#sec-06-intro}

O capítulo anterior construiu layouts a partir do zero — box model, floats,

## 6.1 O Problema que os Frameworks Resolvem {#sec-06-problema}

### 6.1.1 CSS manual a escala {#sec-06-problema-escala}

Num projecto pequeno, escrever CSS do zero é rápido e directo. A partir de
um certo ponto, a escala impõe três problemas estruturais:

**Consistência** — manter os mesmos espaçamentos, tamanhos e cores em dezenas
de componentes (botões, formulários, navbars) exige repetição manual. Uma
alteração ao estilo de um botão pode implicar editar centenas de linhas em
múltiplos ficheiros.

**Responsividade** — escrever `@media` queries para todos os breakpoints de
cada componente é trabalhoso. Vinte componentes com quatro breakpoints cada
equivale a oitenta blocos de media queries para manter.

**Manutenção** — o CSS não tem sistema de tipos nem módulos nativos. CSS não
estruturado cresce para um estado que os programadores designam por *CSS soup*:
um ficheiro difícil de ler, difícil de alterar sem quebrar algo inesperado.

### 6.1.2 O que é um framework CSS {#sec-06-o-que-e}

Um framework CSS é uma biblioteca de estilos, componentes e convenções
prontas a usar — escritas uma vez, reutilizáveis em qualquer projecto. Em
vez de escrever o CSS de cada botão ou navbar do zero, o framework fornece
classes que se aplicam directamente no HTML.

O compromisso é claro: **ganha-se velocidade e consistência**, perde-se algum
controlo fino sobre o CSS gerado. Na maioria dos projectos, este compromisso
é favorável.

### 6.1.3 Panorama dos frameworks principais {#sec-06-panorama}

| Framework | Abordagem | Ponto forte | Caso de uso típico |
|:----------|:----------|:------------|:------------------|
| **Bootstrap** | Componentes | Curva baixa, comunidade enorme | Projectos académicos, prototipagem, portais |
| **Tailwind CSS** | Utility-first | Controlo total, performance | Design systems, aplicações modernas |
| **Foundation** | Componentes | Grid avançado, SASS nativo | Projectos enterprise |
| **Bulma** | Componentes | CSS puro (sem JS), leve | Projectos ligeiros |
| **Materialize** | Material Design | Animações incluídas, estética Google | Apps com look Android/Google |

::: {.callout-note}
#### Por que Bootstrap nesta UC?
Bootstrap combina curva de aprendizagem baixa, documentação exemplar e
presença massiva no mercado de trabalho. É o ponto de entrada mais eficaz
para um primeiro contacto com frameworks. Os conceitos aprendidos —
grelha, breakpoints, utilitários — transferem-se para qualquer outro
framework.

## 6.2 Bootstrap — Instalação via CDN {#sec-06-cdn}

### 6.2.1 O que é um CDN {#sec-06-cdn-def}

Um **CDN** (*Content Delivery Network*) é uma rede distribuída de servidores
que disponibiliza ficheiros estáticos (CSS, JavaScript, fontes) a partir do
nó geograficamente mais próximo do utilizador. Usar o Bootstrap via CDN
significa não descarregar nem hospedar os ficheiros — o browser carrega-os
directamente dos servidores do Bootstrap.

**Vantagens:**

- Configuração imediata — sem instalação, sem npm, sem build step
- O ficheiro pode estar em cache no browser do utilizador (se já visitou
  outro site com o mesmo CDN)
- Sempre actualizado para a versão especificada

**Desvantagens:**

- Dependência de conectividade — não funciona offline
- Impossível personalizar (alterar variáveis SASS, etc.)
- Produção: preferível servir os ficheiros do próprio servidor

### 6.2.2 Estrutura mínima de uma página Bootstrap {#sec-06-estrutura-pagina}

Uma página HTML com Bootstrap requer três elementos essenciais:

```html
<!DOCTYPE html>
<html lang="pt">
<head>
  <meta charset="UTF-8">

  <!-- 1. viewport meta — essencial para responsividade -->
  <meta name="viewport" content="width=device-width, initial-scale=1">

  <!-- 2. Bootstrap CSS — deve vir antes de qualquer CSS personalizado -->
  <link
    href="https://cdn.jsdelivr.net/npm/bootstrap@5.3.3/dist/css/bootstrap.min.css"
    rel="stylesheet">

  <title>Página com Bootstrap</title>
</head>
<body>

  <!-- conteúdo aqui -->
  <div class="container">
    <h1>Olá, Bootstrap!</h1>
  </div>

  <!-- 3. Bootstrap JS Bundle — no final do body, antes de </body> -->
  <!-- inclui Popper.js para tooltips, dropdowns e modals -->
  <script
    src="https://cdn.jsdelivr.net/npm/bootstrap@5.3.3/dist/js/bootstrap.bundle.min.js">
  </script>

</body>
</html>
```

::: {.callout-note}
#### CSS antes do JS
O CSS deve ser incluído no `<head>` para que o estilo seja aplicado antes
de o conteúdo ser renderizado. O JS é colocado no final do `<body>` para
não bloquear o carregamento da página. O `bootstrap.bundle.min.js` inclui
já o Popper.js — não é necessário adicioná-lo separadamente.
:::

### 6.2.3 CSS personalizado sobre Bootstrap {#sec-06-css-proprio}

Ao adicionar CSS personalizado, este deve ser incluído **depois** do CSS
do Bootstrap para que as regras próprias possam sobrepor as do framework:

```html
<head>
  <!-- 1. Bootstrap CSS -->
  <link href="bootstrap.min.css" rel="stylesheet">

  <!-- 2. CSS próprio — vem depois, tem precedência -->
  <link href="styles.css" rel="stylesheet">
</head>
```

## 6.3 O Sistema de Grid Bootstrap {#sec-06-grid}

### 6.3.1 Container, row e col {#sec-06-grid-crc}

O grid Bootstrap assenta em três classes hierárquicas que espelham
exactamente a grelha de 12 colunas estudada no capítulo anterior — a
diferença é que o Bootstrap gere o clearfix, os gutters e a
responsividade automaticamente.

```html
<div class="container">         <!-- centra e define largura máxima -->
  <div class="row g-3">         <!-- linha de grelha; g-3 = gutter de 1rem -->
    <div class="col-8">         <!-- ocupa 8 das 12 colunas -->
      Conteúdo principal
    </div>
    <div class="col-4">         <!-- ocupa 4 das 12 colunas -->
      Sidebar
    </div>
  </div>
</div>
```

o espaço restante igualmente entre todos os `col` da linha.

### 6.3.2 Os 6 breakpoints do Bootstrap {#sec-06-breakpoints}

O Bootstrap define seis pontos de quebra, todos com `min-width` —
confirmando que o Bootstrap é **mobile-first por definição**. Abaixo de
576px não existe media query; o CSS base aplica-se a todos os tamanhos.

| Nome | Prefixo | Condição | Container |
|:----:|:-------:|:--------:|:---------:|
| xs | *(nenhum)* | < 576px — sem query | 100% |
| sm | `sm` | ≥ 576px | 540px |
| md | `md` | ≥ 768px | 720px |
| lg | `lg` | ≥ 992px | 960px |
| xl | `xl` | ≥ 1200px | 1140px |
| xxl | `xxl` | ≥ 1400px | 1320px |

::: {.callout-note}
#### Container ≠ Breakpoint
O container é sempre mais estreito que o breakpoint — para `md` (≥ 768px),
o container tem 720px. Os 48px restantes (24px de cada lado) formam as
margens automáticas que centram o conteúdo. O `container-fluid` é a
excepção: ocupa sempre 100% da largura.
:::

### 6.3.3 Classes responsivas — col-{bp}-{n} {#sec-06-responsivo}

A responsividade Bootstrap é **declarativa**: especifica-se no HTML como
cada coluna se comporta a cada breakpoint, sem escrever uma única media
query:

```html
<!-- Uma coluna que se adapta automaticamente:
     mobile (< 768px): linha completa
     tablet (≥ 768px): meia largura
     desktop (≥ 992px): um terço da largura -->

<div class="row g-3">
  <div class="col-12 col-md-6 col-lg-4">Card A</div>
  <div class="col-12 col-md-6 col-lg-4">Card B</div>
  <div class="col-12 col-md-6 col-lg-4">Card C</div>
</div>
```

As classes aplicam-se de forma **cumulativa** — `col-md-6` aplica-se em md
*e em todos os breakpoints maiores*, até ser sobreposta por `col-lg-4`.

## 6.5 Utilitários Bootstrap {#sec-06-utilitarios}

Os utilitários são classes atómicas que substituem CSS de uma linha —
aplicadas directamente no HTML, sem escrever regras adicionais.

### 6.5.1 Spacing — margin e padding {#sec-06-spacing}

O padrão é `{tipo}{lado}-{escala}`, onde o tipo é `m` (margin) ou `p`
(padding), o lado é opcional (`t`, `b`, `s`, `e`, `x`, `y`), e a escala
vai de 0 a 5 (em múltiplos de `0.25rem`) ou `auto`.

```html
<div class="mt-3">   <!-- margin-top: 1rem -->
<div class="mb-2">   <!-- margin-bottom: 0.5rem -->
<div class="px-4">   <!-- padding horizontal: 1.5rem -->
<div class="py-2">   <!-- padding vertical: 0.5rem -->
<div class="m-0">    <!-- margin: 0 em todos os lados -->
<div class="mx-auto"> <!-- margin horizontal: auto → centrado -->
```

| Escala | Valor rem | Valor px (base 16px) |
|:------:|:---------:|:--------------------:|
| 0 | 0 | 0 |
| 1 | 0.25rem | 4px |
| 2 | 0.5rem | 8px |
| 3 | 1rem | 16px |
| 4 | 1.5rem | 24px |
| 5 | 3rem | 48px |

### 6.5.2 Display, Flexbox e texto {#sec-06-utilitarios-outros}

```html
<!-- Display -->
<div class="d-none">    <!-- display: none -->
<div class="d-flex">    <!-- display: flex -->
```

## 6.6 Micro-interações em CSS Puro {#sec-06-micro}

### 6.6.1 O Bootstrap já anima — mas como? {#sec-06-micro-bootstrap}

Se inspeccionarmos o CSS do Bootstrap com as DevTools do browser
(**F12 → Sources → bootstrap.min.css**), encontramos isto:

```css
/* Bootstrap internals — transition nos botões */
.btn {
  transition: color .15s ease-in-out,
              background-color .15s ease-in-out,
              border-color .15s ease-in-out,
              box-shadow .15s ease-in-out;
}

/* Transição no collapse da navbar */
.collapsing { transition: height .35s ease; }

/* Fade nos modals */
.fade { transition: opacity .15s linear; }
```

O efeito de hover dos botões, a abertura suave da navbar, o fade dos modals
— nada disto é JavaScript. É CSS puro. O Bootstrap escreve as transições por
nós; as secções seguintes ensinam a fazê-lo por conta própria, em qualquer
contexto, com ou sem framework.

### 6.6.2 transition {#sec-06-transition}

A propriedade `transition` suaviza a mudança de qualquer propriedade CSS
ao longo do tempo. A sintaxe é:

```css
transition: propriedade duração easing atraso;
```

```css
/* Exemplo simples: mudar a cor de fundo em 0.3s */
.btn {
  background: #0d6efd;
  transition: background-color 0.3s ease;
}
.btn:hover { background: #0a58ca; }

/* Múltiplas propriedades */
.card {
  transform: translateY(0);
  box-shadow: none;
  transition: transform 0.2s ease,
              box-shadow 0.2s ease;
}
.card:hover {
  transform: translateY(-6px);
  box-shadow: 0 8px 24px rgba(0,0,0,.15);
}

/* ⚠ Evitar: transition: all — anima tudo, impacto de performance */
```

Os quatro parâmetros de `transition`:

| Parâmetro | Valores | Descrição |
|:----------|:--------|:----------|
| `propriedade` | `color`, `opacity`, `transform`, … | O que muda. Preferir propriedades específicas a `all`. |
| `duração` | `0.3s`, `300ms` | Quanto tempo dura a transição. |
| `easing` | `ease`, `linear`, `ease-in`, `ease-out`, `cubic-bezier()` | Curva de velocidade. |
| `atraso` | `0s`, `0.1s` | Espera antes de começar. Opcional. |

### 6.6.3 transform {#sec-06-transform}

A propriedade `transform` modifica a aparência visual de um elemento —
movendo-o, escalando-o ou rodando-o — **sem alterar o fluxo do documento**.
Os elementos vizinhos não se deslocam. O browser usa aceleração GPU, o que
torna as animações com `transform` mais eficientes do que animar `top` ou
`left`.

```css
/* Mover */
.el:hover { transform: translateY(-8px); }   /* desloca para cima 8px */
.el:hover { transform: translateX(4px); }    /* desloca para a direita */
.el:hover { transform: translate(-4px, -4px); } /* ambos os eixos */

/* Escalar */
.el:hover { transform: scale(1.05); }        /* 5% maior */
.el:hover { transform: scale(0.95); }        /* 5% menor */

/* Rodar */
.arrow:hover { transform: rotate(90deg); }   /* sentido horário */
.icon.open   { transform: rotate(45deg); }   /* ícone × a partir de + */

/* Combinar — sempre numa única declaração */
.card:hover {
  transform: translateY(-4px) scale(1.02);   /* ✓ */
}
/* ✗ Errado: duas declarações transform sobrepoem-se */
.card:hover { transform: translateY(-4px); }
.card:hover { transform: scale(1.02); }      /* cancela a anterior */
```

### 6.6.4 Pseudo-classes como gatilhos {#sec-06-pseudo}

As pseudo-classes definem **estados** de um elemento — são o mecanismo que
activa as micro-interações sem JavaScript.

```css
.btn {
  background: #0d6efd;
  color: white;
  transition: background 0.2s ease, transform 0.1s ease;
}

/* Estado: rato sobre o elemento */
.btn:hover {
  background: #0a58ca;
  transform: translateY(-2px);
}

/* Estado: elemento pressionado */
.btn:active {
  transform: translateY(0);   /* "afunda" no clique */
}

/* Estado: elemento com foco (teclado ou clique) */
.btn:focus {
  outline: 3px solid rgba(13,110,253,.4);
  outline-offset: 2px;
}

/* Estado: foco apenas por teclado (não por clique) */
/* mais limpo que :focus para utilizadores de rato */
.btn:focus-visible {
  outline: 3px solid #0d6efd;
  outline-offset: 2px;
}
```

::: {.callout-note}
#### :focus e acessibilidade
Nunca remover o outline de foco sem fornecer uma alternativa visual.
O outline é o único indicador de posição para utilizadores que navegam
por teclado (pessoas com limitações motoras). O `:focus-visible` permite
mostrar o outline apenas durante navegação por teclado, sem o mostrar
a utilizadores de rato — o melhor dos dois mundos.
:::

### 6.6.5 @keyframes — animações autónomas {#sec-06-keyframes}

Enquanto `transition` reage a uma mudança de estado (hover, focus), o
`@keyframes` define uma sequência de estados que corre automaticamente,
independentemente da interação do utilizador.

A utilização é um processo em dois passos:

**Passo 1 — Definir a animação** com `@keyframes`:

```css
/* Sintaxe with from/to */
@keyframes fadeInUp {
  from {
    opacity: 0;
    transform: translateY(20px);
  }
  /* ponto intermédio opcional: */
  /* 50% { opacity: 0.5; }      */
  to {
    opacity: 1;
    transform: translateY(0);
  }
}

/* Sintaxe com percentagens — permite múltiplos estados */
@keyframes pulse {
  0%, 100% { transform: scale(1); }
  50%       { transform: scale(1.05); }
}
```

**Passo 2 — Aplicar** com a propriedade `animation`:

```css
/* Shorthand: nome duração easing atraso iterações direção fill-mode */
.hero    { animation: fadeInUp 0.6s ease-out; }
.badge   { animation: pulse 2s ease-in-out infinite; }

/* Propriedades individuais */
.card {
  animation-name:            fadeInUp;
  animation-duration:        0.4s;
  animation-timing-function: ease-out;
  animation-delay:           0.1s;
  animation-fill-mode:       forwards; /* mantém o estado final */
}
```

::: {.callout-note}
#### animation-fill-mode: forwards
Por omissão, quando uma animação termina, o elemento volta ao estado
inicial. `fill-mode: forwards` mantém o estado final da animação —
essencial para efeitos de entrada como `fadeInUp`, onde queremos que
o elemento permaneça visível depois de aparecer.
:::

### 6.6.6 Boas práticas e acessibilidade {#sec-06-boas-praticas}

```css
/* ✓ Especificar propriedades — não usar transition: all */
.btn { transition: background-color 0.2s ease, transform 0.1s ease; }

/* ✓ Preferir transform e opacity — aceleração GPU, sem reflow */
/* ✗ Evitar animar top, left, width, height — causam reflow */

/* ✓ Respeitar prefers-reduced-motion */
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration:   0.01ms !important;
    transition-duration:  0.01ms !important;
  }
}

/* ✓ Nunca remover :focus sem alternativa visível */
/* ✗ .btn:focus { outline: none; }  — inacessível */
/* ✓ .btn:focus-visible { outline: 3px solid #86b7fe; } */

/* ✓ Animar com propósito — cada micro-interação deve comunicar algo:
      hover → "isto é clicável"
      active → "o clique foi registado"
      fadeInUp → "novo conteúdo chegou"      */
```

## Síntese do Capítulo {#sec-06-sintese}

Os **frameworks CSS** resolvem o problema da consistência e da
responsividade a escala. O Bootstrap é o mais adoptado: inclui via CDN,
usa classes no HTML, e encapsula as media queries, o grid e os componentes
mais comuns.

O **grid Bootstrap** é o sistema de 12 colunas do capítulo anterior,
automatizado e responsivo. Os seis breakpoints (`xs` a `xxl`) usam todos
`min-width` — o Bootstrap é mobile-first por definição. As classes
`col-{bp}-{n}` aplicam-se a partir do breakpoint indicado e são cumulativas.

Os **componentes** (navbar, cards, buttons, forms) e os **utilitários**
(spacing, display, flex, texto, cores) eliminam CSS repetitivo. A regra
de ouro é incluir o CSS próprio **depois** do Bootstrap para poder
sobrepor estilos.

As **micro-interações** — `transition`, `transform`, pseudo-classes e
`@keyframes` — são CSS puro, independentes do Bootstrap. O Bootstrap
usa-as internamente; aprendê-las em CSS nativo significa usá-las em
qualquer projecto. `transform` e `opacity` são as propriedades a animar:
usam GPU, não causam reflow, e produzem transições fluidas. Respeitar
`prefers-reduced-motion` e nunca remover `:focus` sem alternativa
são requisitos de acessibilidade, não opções.
